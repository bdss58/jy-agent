## 2026-05-01 00:39 [ship]
Shipped `fix(tools): bound run_shell output to prevent OOM kill` (commit 79a8e98).

Background: 2026-04-30 23:31 macOS jetsam SIGKILLed jy-agent Python (PID 83341, ~74 GiB resident). Codex correctly diagnosed: `subprocess.run(capture_output=True, text=True)` in `jyagent/tools/core.py::run_shell` buffered the entire child stdout+stderr in RAM before truncating to 50 000 chars. A runaway command could push the parent into multi-GB territory.

Fix design (mirrors the existing background path's discipline):
- New `_BoundedStreamReader(threading.Thread)` drains a binary pipe into bounded head (8 KB) + tail (128 KB) bytearrays.
- New `_drain_proc_bounded(proc)` wires up two readers (stdout, stderr) with `on_overflow=SIGKILL pgroup` at 32 MB per stream.
- Unified the legacy and cooperative-cancel paths into a single `Popen + drain + poll` flow. `_cancel_event` is now just a poll-loop predicate.
- `text=False` (bytes) — decode at `.collect()` time, cuts memory ~2× vs the str-decode-everything path.
- Defensive: keep draining after overflow fires so child's `write()` doesn't block on a full pipe; SIGKILL still works.

Bug found during testing: 50 000-char user-visible cap was applied AFTER appending the overflow marker → marker got chopped off when output filled the cap. Fixed by reordering: cap stdout/stderr first, then append overflow marker (always visible).

Tests: 11 new in `tests/test_run_shell_bounded.py` — RSS-bound (500 MB stdout → parent grows <100 MB), overflow tagging, stderr-only overflow, cancel still works, helper unit tests. Full suite: 734 passed.

Files: `jyagent/tools/core.py` (+208 / -57), `tests/test_run_shell_bounded.py` (new, 225 lines).

## 2026-05-01 01:26 [debug]
Found and fixed a head/tail boundary bug in `_BoundedStreamReader.collect()` (commit f5a5b2c) while auditing what the LLM actually sees from bounded tool results.

Pre-fix logic:
    truncated = self._total > (len(self._head) + len(self._tail))
    if not truncated: return bytes(self._tail)

The "not truncated" branch is correct only when `total <= tail_max`. When `tail_max < total <= head_max + tail_max`, the tail buffer has already dropped its early bytes (head still has them), but the code returned tail-only — silently losing the head bytes.

Default config (head=8 KB, tail=128 KB) → buggy band (128 KB, 136 KB]. After the 50 K char cap, the LLM saw an arbitrary middle slice (~[total-128K, total-78K]) instead of the start.

Fix: three explicit regimes:
  1. total ≤ tail_max         → return tail
  2. total ≤ head_max+tail_max → head[:total-tail_max] + tail (no marker, no loss)
  3. otherwise                 → head + elision marker + tail

5 new tests, all 3 regime-2 tests fail against pre-fix (verified by stashing). Full suite 742 passed.

Pattern lesson: when implementing head+tail bounded buffers, the "is the data complete" condition is `total <= tail_max`, NOT `total <= head + tail`. The latter is when there's no GAP, which is different.

## 2026-05-01 01:47 [ship]
Shipped commit 075b943: spill-to-disk + env-configurable caps for run_shell.

Closes the partial-context gap with Claude Code by making large-output recovery possible, while keeping our head+tail inline preview (better than Claude Code's head-only ~2 KB preview, per issue #17944).

Design highlights:
1. **Lazy spill activation** at exactly the right moment: when next chunk would push total > tail_max. At that instant, tail buffer still holds bytes [0, total) — dump it to /tmp, then tee every subsequent chunk. Memory cost stays O(head + tail), no extra pre-spill buffer needed.
2. **Spill survives SIGKILL**: when overflow kills the child at 32 MB, spill keeps writing — file captures ~all pre-kill bytes.
3. **Factory pattern** for spill path: tests inject a mock factory; production factory creates `/tmp/jyagent_runshell_<out|err>_*.out` via mkstemp. Factory raising = silent fallback (read-only rootfs etc).
4. **Env-var config** with explicit stderr warnings on malformed values — defends against Claude Code's #17944 regression where BASH_MAX_OUTPUT_LENGTH was silently ignored after v2.1.2. Knobs: HEAD_BYTES, TAIL_BYTES, HARD_KILL_BYTES, OUTPUT_CHAR_CAP, SPILL (bool), SPILL_DIR.

## 2026-05-06 13:03 [session]
Bug #2 fix: log/view conflation in jy-agent session. Co-worked with Codex (codex review --uncommitted).

**Architecture shipped**:
- New jyagent/memory/event_log.py — append-only per-session JSONL at data/sessions/events/<sid>.jsonl. EventLog class with emit/emit_many (batched fsync), get_events (positional slice), monotonic seq.

- Snapshot payload now includes `last_event_seq` for sync-detection.
- compaction emits a `{"kind":"compaction", "drop_count", "replacement_messages", ...}` event before the swap, preserving pre-compaction context AND the synthetic summary+file-reinjection messages.
- replay_from_events(sid) reconstructs the live view from the log alone (walks message events, applies compaction transformations).

**Codex's pre-review (planning phase) caught**:
1. My crash-analysis was wrong — extend()→crash before checkpoint loses both views, not just one. Per-turn boundary IS the durability unit.
2. Compaction event MUST embed replacement_messages (file-reinjection content not reconstructible from filesystem at replay time). Initial plan only had summary.
3. Use drop_count not range — unambiguous.
4. Batch fsync at checkpoint boundary, not per-event. emit_many() does one fsync per batch.
5. Event schema: `{"kind":"message","message":<full dict>}` not flattened — preserves tool_call_id, tool_name, is_error, etc.
6. Lazy attach — ConversationMemory()/clear() must NOT touch logs.
