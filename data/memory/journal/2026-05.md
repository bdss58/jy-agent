## 2026-05-18 00:48 [debug]
Fixed the pre-existing test-ordering bug in the three memory test files (referenced in 9d1f0aa journal).

Root cause: each of `test_memory_upgrades.py`, `test_memory_phase1.py`, and `test_memory_tiers.py` patches `jyagent.config.MEMORY_DIR / TOPICS_DIR / JOURNAL_DIR / MEMORY_MD_FILE` at MODULE-IMPORT time. Pytest imports all test modules up-front BEFORE running any test, so all three module bodies execute and the LAST-imported file wins. When the user passes the three files in `upgrades phase1 tiers` order, config ends up pointing at tiers' tmpdir, but upgrades' tests run FIRST against that wrong tmpdir, causing cross-file state accumulation and 6 deterministic failures.

Demo:
  import upgrades → config.MEMORY_MD_FILE = upgrades_tmp/...
  import phase1   → config.MEMORY_MD_FILE = phase1_tmp/...
  import tiers    → config.MEMORY_MD_FILE = tiers_tmp/...
  run upgrades tests → operate on tiers_tmp (wrong!)
  upgrades' setup() rmtree's upgrades_tmp (no-op, irrelevant dir)
  state leaks across upgrades tests inside tiers_tmp

Fix: minimal. In each file, prepend the four `config.X = os.path.join(_tmpdir, ...)` lines to the existing `setup()` helper, so every test that calls `setup()` (most disk-touching tests) re-asserts config correctness for ITS file's `_tmpdir`. Zero call-site changes. The ~12 pure-function tests per file that don't call setup() don't touch disk and don't care.

Verification: ran 4 different file orderings of (upgrades, phase1, tiers) → all 132/132 pass. Full suite: 1038 passed / 6 skipped / 0 failed.

Lesson: never patch module-level globals at test-module-import time when multiple test files patch the SAME globals. Either (a) re-assert per-test (chosen here, minimal change), or (b) use conftest.py + autouse fixtures (cleaner long-term but bigger diff). Document the rationale next to the setup() helper so the next reader doesn't innocently revert it.
